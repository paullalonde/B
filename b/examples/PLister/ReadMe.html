PLister Sample

The sample demonstrates the following techniques:

  • Multiple-document application.
  • Simple document file I/O.
  • Strict separation between business logic and user interface (MVC-like).
  • Factored design. The UI sends Apple Events to the application's data model objects in order to perform any modification on the model.
  • Undo.  All user actions in PLister are undoable.
  • Integration of Apple Events and Undo.  Event commands sent via Apple Events are undoable.
  • DataBrowser support for drag & drop, help tags, and the Services menu.
  • Saving & restoring the databrowser's configuration (column location and size).
  • Use of std::istream & std::ostream to read/write drag and scrap contents.
  • Localisation. PLister's UI is available in English and French.

The main classes are described next.

ApplicationDocPolicy

The document policy class. This class derives from B::MultipleDocumentPolicy<B::MultipleDocumentFactory>, which is a long way of saying that it supports multiple open documents of multiple kinds.

Application

The main application class. This class derives from B::Application<ApplicationDocPolicy>, B::PrintPolicy>.

Browser

This class derives from B::DataBrowser and adds a number of PLister-specific commands.

BrowserItem

This class derives from B::DataBrowserItem and represents one line in the browser.  Each such line corresponds to a single ModelItem.

Document

This class represents an open property list document.  It derives from B::Document<B::NullPrintPolicy>, which means that it doesn't support printing.

ModelItem

This class represents one element of a property list.  An element is a dictionary, array, date, integer, boolean, real, or blob.

Window

This class derives from B::Window and adds a number of PLister-specific commands.
